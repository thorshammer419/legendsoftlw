# Game Design Decisions

Complete record of all design decisions made for The Legends of TLW.

---

## Core Concept
- Shared party D&D 5e RPG web app
- LLM (GPT-4.1) acts as Dungeon Master — no human GM
- Group of players share a single story in a web interface
- D&D 5e SRD 5.1 rules (CC-BY 4.0) — NOT 2024 revised ruleset
- Original world lore generated by LLM in SRD style

## Players & Campaigns
- Any player can create a campaign (no admin-only creation)
- Campaign creator is automatically the admin
- Admin can promote other players to admin role
- Players name their own adventuring party per campaign
- No limit on how many campaigns a player can be in simultaneously
- No hard limit on players per campaign — admin can set a limit per campaign
- Initial test group (The Lord's Wrath): 3-9 players, max 8-9

## Round Mechanics
- Each player submits their own intended action per round
- Round resolves when ALL players have submitted OR timeout expires
- Timeout is configurable per campaign (can be disabled entirely)
- Absent players receive a graceful narrative skip
  ("Thorin hesitates, frozen by indecision...")
- No player is ever permanently removed from a campaign

## Schedule System
- Admin configures per campaign:
  - Timeout duration (hours) or disabled
  - Quiet hours (start time, end time, timezone)
  - Active days of week
  - Blackout dates
- Default: 24hr timeout, quiet hours 9pm-9am CST, all days active
- Timer pauses during quiet hours, resumes after

## Inactivity System
- Auto-inactivity threshold: 2 consecutive combat encounters OR
  4 consecutive scenes without submitting an action
- Admin can manually toggle any player to inactive at any time
- Admin can configure thresholds per campaign
- Inactive players:
  - Excluded from combat narratives
  - Not critical characters in new story chapters
  - LLM explicitly told not to interact with them
  - Still gracefully skipped each round
  - Character preserved in Cosmos DB permanently
- Reactivation: admin toggles back to active
  → cinematic re-introduction narrative generated (same as new player joining)
- When ALL players are inactive: campaign auto-pauses
  → Weekly reminder email sent to all players
  → Resumes from exact state when any player reactivates
  → GPT-4.1-mini generates "welcome back" narrative on resume

## Character System
- Full D&D 5e SRD 5.1 character creation
- Hybrid flow: React web form → Azure Functions → Cosmos DB
- Unique link per player maps to their email (canonical ID)
- Character death: SOFT DEATH only — characters never permanently die
  (drop to 0 HP, stabilize eventually — no death saving throw failure)

## Action System

### Mode 1: Character Sheet Actions
- Complete ability/spell list displayed in UI like a real character sheet
- Organized into: Actions, Bonus Actions, Reactions, Spells (by slot level)
- Unavailable actions greyed out based on:
  - Spent spell slots
  - Active conditions (e.g. Silenced = verbal spells greyed out)
  - Used action economy (used Action this round = Action greyed out)
- Player selects ability → UI renders ability-specific interface:
  - Target selection (populated from active_npcs + party members)
  - Any required pre-roll declarations
  - Automatic dice rolling (all required dice roll automatically)

### Mode 2: Freeform Text
- Player describes intended action in text box
- GPT-4.1-mini pre-validates privately in action panel (back-and-forth)
- DM prompts required rolls
- Player rolls dice as prompted
- Player formally submits action WITH roll results attached
- If invalid: DM explains why, player tries again
- Round timer continues during freeform dialogue

### Hybrid Contextual Action List
- Static always-available actions (Attack, Dodge, Dash, etc.) defined in JSON config
- GPT-4.1-mini generates up to 5 situational additions per player per round
- Pre-generated after each round resolves, cached in Cosmos DB
- Rendered as buttons alongside static list

## Dice System
- All 7 dice available: d4, d6, d8, d10, d12, d20, d100
- Players can select any number and combination to roll
- Cryptographic randomness: `crypto.getRandomValues()`
- Character sheet actions: all required dice roll automatically
- Freeform actions: DM instructs which dice to roll, player selects and rolls
- Roll results displayed in narrative feed

## NPC System
- Major NPCs: persistent Cosmos DB documents
  - Full stat blocks, motivations, relationships
  - Per-player disposition tracking (hostile/neutral/friendly/etc.)
  - Interaction log per round
  - GPT-4.1-mini auto-creates document when named NPC first appears
  - Admin can seed NPCs before campaign starts
  - Only last 3 interactions injected into LLM prompt (token efficiency)
- Minor NPCs: LLM managed only (no Cosmos DB document)

## Player Joining Mid-Campaign
- Any player can join at any time via invite link
- On joining:
  1. Private catch-up summary generated (GPT-4.1-mini) — shown only to new player
  2. Cinematic introduction narrative generated (GPT-4.1) — broadcast to all players
     Style: "a character being introduced in a fantasy novel"
- Introduction narrative stored in narrative summary for continuity
- New player's character enters next round

## Campaign Completion
- Two modes set at campaign creation:
  - Finite: LLM declares complete when main quest resolves
            Admin can also end manually at any time
  - Ongoing: After each quest completes, LLM generates new quest
             Admin can still end manually
- On completion: campaign archived permanently
- Archived campaigns:
  - Players can read full narrative history
  - Can export as LLM-generated fantasy novel (PDF)
  - Can start a sequel campaign

## Novel Export
- GPT-4.1 rewrites full campaign narrative as polished fantasy novel
- Organized into chapters based on major story arcs
- Style: published fantasy novel (Tolkien, Sanderson, Martin)
- No game mechanics references in output
- Includes epilogue for each character
- Generates evocative chapter titles
- PDF generated via Azure Functions
- Stored in Azure Blob Storage
- Download link emailed to all players

## Sequel Campaigns
- Players choose: continue same character OR create new character
- Continuing characters: level up based on XP, HP restored, equipment kept
- `legend` field in new campaign document stores:
  - Previous campaign summary (max 500 tokens)
  - Key events list
  - Returning vs new character list
- LLM opening narrative references previous campaign events naturally

## Notifications
- Per player, per campaign preferences:
  - Email (Azure Communication Services)
  - Browser push notifications (Web Push + VAPID keys)
- Triggers:
  - New round started → notify all active players
  - Round resolving in 2 hours → reminder to players who haven't submitted
  - Player marked inactive → notify that player
  - Player reactivated → notify that player
  - New player joined → notify all active players
  - Campaign paused/resumed → notify all players
  - Novel export ready → email download link to all players
- 2hr warning suppressed during quiet hours
- Notifications respect schedule settings

## Authentication
- Azure Static Web Apps built-in social login
- Providers: Google, Facebook, Microsoft
- Email address is canonical player identifier (provider-agnostic)
- Stored as partition key on player document in Cosmos DB

### Mobile (story-first with drawers)
- Narrative feed always visible and primary
- Character sheet: left drawer (swipe or hamburger)
- Quest log + party status: right drawer
- Action panel: pinned to bottom of screen always

## Admin Panel
- In-game drawer (quick access):
  - Toggle player active/inactive
  - View inactivity stats per player
  - Pause/resume campaign
- Separate admin page (full settings):
  - Campaign name, theme, party name
  - Schedule configuration (timeout, quiet hours, active days, blackout dates)
  - Inactivity thresholds
  - Max player limit
  - Campaign type (finite vs ongoing)
  - NPC management (view, create, edit major NPCs)
  - Player management (roles, status)
  - End campaign manually
  - Start sequel campaign
  - Export campaign novel

## RAG System
- Sources:
  - SRD 5.1 markdown (section-chunked, ~500 tokens, include heading context)
  - 5e-database JSON (one document per entity: spell, monster, class, race, etc.)
- Metadata on every document: category, source, tags
- Categories: spell | monster | class | rule | equipment | condition
- Retrieval triggered by GPT-4.1-mini RAG query generation (max 5 queries/round)
- Filtered by category at search time for precision
- Retrieved chunks injected into GPT-4.1 narrative prompt

## Cost Controls
- Monthly Azure spend cap configured in Azure
- Admin notified at 80% of cap
- All campaigns pause gracefully at 100% of cap
- All players notified with explanation
- Resumes next billing cycle
